# 项目启动四份核心文档初稿

> 适用对象：基于“兼容 Legado 书源格式与主流规则能力”的多端本地客户端项目。
>
> 当前建议路线：**统一 Core，iOS / Android / Windows 做壳**；以**兼容书源格式与行为**为目标，不直接复用 Android 实现代码。

---

# 一、《项目立项边界》

## 1. 文档目的
本文件用于明确项目的目标边界、实现路线、许可证策略与阶段性范围，避免团队在开发过程中因目标漂移、兼容口径不一致或技术路线分歧而导致返工。

## 2. 项目定义
本项目旨在开发一套可在 **iOS / Android / Windows** 上使用的本地客户端体系，其核心能力是：

- 兼容 Legado 书源 JSON 的主流字段结构与规则语义
- 提供统一的书源解析、规则执行、网络请求、缓存与调试能力
- 在不同平台上复用统一内核，平台端主要负责 UI、系统能力接入和本地交互

## 3. 项目要做什么
本项目首要建设以下能力：

1. **统一书源兼容内核**
   - 统一解析 BookSource 结构
   - 统一实现搜索、目录、正文等规则执行流程
   - 统一错误码、日志口径与失败分类

2. **多端本地客户端**
   - iOS 客户端
   - Android 客户端
   - Windows 客户端

3. **兼容性与可维护性资产**
   - 样本库
   - 兼容矩阵
   - 回归测试体系
   - 调试与诊断能力

4. **安全与合规控制**
   - 登录态与 Cookie 管理
   - JS 规则降级与后续受控支持
   - 用户协议、隐私政策、内容声明、投诉处理路径

## 4. 项目不做什么
当前阶段明确不做以下内容：

- 不直接复用、翻译或移植 Legado Android 代码
- 不承诺首版完整支持所有历史书源与全部边缘规则
- 不提供内置内容平台或内置书源市场
- 不以首版目标支持复杂在线调试服务、远程 Web 服务生态复刻
- 不优先做跨设备云同步、账号体系、社区分享体系

## 5. 技术路线决策
### 5.1 总体路线
本项目采用：

**统一 Core + 多端壳层**

即：
- Core 层负责书源兼容、规则执行、网络与缓存能力
- 各平台负责 UI、阅读交互、平台系统集成

### 5.2 不采用方案
当前不采用以下路线：
- 各端独立实现书源逻辑
- 先做单端阅读器再临时抽取共用逻辑
- 直接围绕 UI 先行开发，再回补内核能力

## 6. 许可证与 clean-room 原则
### 6.1 原则
本项目采取：

**兼容格式与行为，不复用实现代码**

### 6.2 约束
- 禁止直接复制、翻译、改写 Legado Android 源码进入本项目
- 允许参考公开字段定义、接口语义、规则行为与样本表现
- AI 生成代码不得以外部 GPL 实现代码为直接改写来源
- 所有关键实现需通过项目自己的样本与行为测试验证

### 6.3 审查要求
- PR 中如涉及兼容行为实现，需说明依据是“规则行为”还是“样本观察”
- 新增第三方依赖必须附许可证说明
- 审查中发现可疑代码来源时，必须回退并重写

## 7. 首版范围
首版目标为：

- 打通统一 Core
- 建立书源导入、搜索、目录、正文主链路
- 支持非 JS 主路径
- 提供基础 Header、Cookie、缓存与错误定位能力
- 提供最小调试页

## 8. 二期范围
- 逐步放开 JS 支持
- 登录态增强
- coverDecodeJs / loginCheckJs 等增强能力
- RSS / 订阅源补全
- 本地 TXT / EPUB 增强
- 调试工具增强

## 9. 成功标准
项目启动阶段完成的标准：

- 已确认统一 Core 路线
- 已明确许可证边界
- 已冻结第一版兼容规范
- 已建立样本库与兼容矩阵
- 已明确首版能力范围与降级口径
- 已建立工程门禁与回归规则

# 二、《书源兼容规范》

## 1. 文档目的
定义本项目对 BookSource 的兼容口径，明确字段支持范围、规则结构、未知字段处理方式与版本扩展策略。

## 2. 兼容目标
本项目兼容目标为：

- **兼容 Legado 书源 JSON 的主流字段结构与主流程行为**
- 保证书源可导入、可解析、可执行主流程
- 尽量做到导入不丢字段、执行不因未知字段崩溃

## 3. 兼容级别定义
### Level A：完整主路径兼容
- 可完成导入
- 可完成搜索
- 可完成目录解析
- 可完成正文解析
- 结果与预期结构一致

### Level B：主路径兼容，部分能力降级
- 可完成导入
- 主链路可运行
- 部分增强能力降级或跳过
- 不影响核心阅读流程

### Level C：可导入但不可完整执行
- 可识别结构
- 可提示不支持字段或规则
- 不崩溃
- 可用于后续补齐支持

### Level D：不兼容
- 字段结构无法识别
- 核心规则缺失或格式错误
- 无法安全降级

## 4. 字段支持策略
### 4.1 核心字段
首版必须保证识别并正确存储：

- `bookSourceName`
- `bookSourceUrl`
- `bookSourceGroup`
- `bookSourceType`
- `bookUrlPattern`
- `customOrder`
- `enabled`
- `enabledExplore`
- `header`
- `loginUrl`
- `loginUi`
- `enabledCookieJar`
- `searchUrl`
- `exploreUrl`
- `ruleSearch`
- `ruleBookInfo`
- `ruleToc`
- `ruleContent`

### 4.2 可选字段
首版可识别并保留，但允许部分不执行：

- `jsLib`
- `loginCheckJs`
- `coverDecodeJs`
- `respondTime`
- `weight`
- `concurrentRate`
- RSS 相关字段
- 发现页与探索页附加字段

### 4.3 未知字段策略
- 导入时保留未知字段原始内容
- 序列化导出时尽量原样回写
- 未知字段不得导致解析崩溃
- 未知字段默认不参与执行，除非后续版本明确支持

## 5. 规则结构兼容要求
### 5.1 必须兼容的规则类型
- CSS Selector
- XPath
- JSONPath
- Regex
- 替换规则
- Header 注入
- 基础 Cookie Jar

### 5.2 首版可识别但降级的规则能力
- `jsLib`
- `loginCheckJs`
- `coverDecodeJs`
- 复杂登录脚本
- 高级解密逻辑

### 5.3 暂不承诺能力
- 完整复刻全部历史边缘脚本行为
- 所有第三方扩展约定字段
- 平台相关非标准增强能力

## 6. 结构化模型要求
BookSource 模型必须满足：

- 支持字段缺省值
- 支持字段可空
- 支持嵌套规则对象
- 支持未知字段透传
- 支持版本扩展不破坏旧版本导入

## 7. 导入与导出要求
### 导入要求
- 合法 JSON 可导入
- 缺失非关键字段时不崩溃
- 错误输入可返回明确错误码

### 导出要求
- 已识别字段按规范输出
- 未知字段尽量保留
- 不因中间编辑过程丢失额外字段

## 8. 向前兼容策略
- 新字段一律按“未知字段保留”处理
- 新规则类型进入执行前必须先进入兼容矩阵
- 旧版本书源导入不得因新字段报错

## 9. 失败分类
- JSON 结构错误
- 核心字段缺失
- 规则字段格式不合法
- 规则类型不支持
- JS 规则需降级
- 登录能力未实现
- 网络策略不匹配

## 10. 验收要求
兼容规范冻结的最低标准：

- 已列出核心 / 可选 / 未知字段策略
- 已列出规则能力范围
- 已定义兼容等级
- 已定义导入导出与向前兼容口径

# 三、《能力支持矩阵》

## 1. 文档目的
明确项目在首版、二期与后续版本中对各类书源规则能力的支持范围、降级策略与实现优先级。

## 2. 分类标准
- **首版必须**：不支持则主链路不可用
- **二期支持**：首版可不实现，但需有规划
- **可降级**：首版不完整实现，但必须识别并提示
- **暂不承诺**：首版与二期均不保证

## 3. 能力矩阵
| 能力 | 首版状态 | 优先级 | 说明 |
|---|---|---:|---|
| CSS Selector | 首版必须 | P0 | HTML 主路径核心能力 |
| XPath | 首版必须 | P0 | 主流书源常用能力 |
| JSONPath | 首版必须 | P0 | JSON 接口类书源需要 |
| Regex | 首版必须 | P0 | 提取与替换基础能力 |
| Header 注入 | 首版必须 | P0 | 请求兼容基础能力 |
| Cookie Jar | 首版必须 | P0 | 基础会话保持 |
| 登录态基础支持 | 二期支持 | P1 | 首版可限制场景 |
| jsLib | 可降级 | P1 | 首版识别、提示、跳过 |
| loginCheckJs | 可降级 | P1 | 首版不完整执行 |
| coverDecodeJs | 可降级 | P1 | 首版不完整执行 |
| RSS / 订阅源 | 二期支持 | P2 | 非首版主链路 |
| 本地 TXT | 二期支持 | P2 | 可较早补齐 |
| 本地 EPUB | 二期支持 | P2 | 后续扩展 |
| 替换规则 | 首版必须 | P0 | 正文清洗核心能力 |
| 调试页 | 首版必须 | P0 | 问题定位必要能力 |
| 请求/响应查看 | 首版必须 | P0 | 最小调试能力 |
| 规则执行结果查看 | 首版必须 | P0 | 最小调试能力 |
| 错误码展示 | 首版必须 | P0 | 回归与定位基础 |
| 本地 HTTP 调试接口 | 暂不承诺 | P3 | 长期预留 |
| WebSocket 调试接口 | 暂不承诺 | P3 | 长期预留 |
| 高级 JS 沙箱 | 二期支持 | P1 | 需独立 PoC |
| 解密增强能力 | 二期支持 | P2 | 依赖样本驱动 |

## 4. 首版支持原则
首版能力的判断标准：

- 是否直接影响“导入 → 搜索 → 目录 → 正文”主链路
- 是否影响主流非 JS 书源兼容率
- 是否属于调试与回归的最低条件

## 5. 降级原则
对于首版不完整支持的能力，必须做到：

- 可识别
- 不崩溃
- 有明确提示
- 可进入失败分类与兼容矩阵统计

## 6. 进入二期的条件
以下能力进入二期前需满足：

- 已积累足够样本
- 已能归纳主流行为模式
- 已完成安全边界评估
- 已具备最小回归覆盖

## 7. 验收要求
能力支持矩阵完成的标准：

- 所有关键能力已归类
- 首版范围明确
- 降级口径明确
- 二期优先级明确

# 四、《工程门禁与回归规则》

## 1. 文档目的
建立项目统一的开发门禁、PR 审查标准与回归测试规则，保证兼容性资产持续累积，避免 AI 生成代码或快速迭代导致质量失控。

## 2. 适用范围
适用于：

- Core 内核代码
- 平台壳层接入代码
- 兼容模型与规则执行代码
- 调试能力
- 样本与回归资产
- 文档与工程配置

## 3. 基础开发规则
### 3.1 提交规则
每个变更必须：
- 绑定 Issue 或任务编号
- 说明影响范围
- 说明是否影响兼容结果
- 说明是否影响错误码或失败分类
- 说明是否需要回退预案

### 3.2 AI 代码规则
AI 生成代码必须满足：
- 必须人工审查后才能合并
- 必须补充或更新测试
- 必须说明是否新增依赖
- 不得直接搬运外部实现代码
- 涉及兼容逻辑时必须给出样本依据

## 4. PR 门禁
每个 PR 至少包含：

1. 关联的样本或规则类型
2. 回归结果说明
3. 风险说明
4. 回退方案
5. 是否影响兼容率统计
6. 是否影响日志与错误码

未满足以上要求的 PR 不得合并。

## 5. 测试门禁
### 5.1 必跑测试
- 单元测试
- 样本回归测试
- 关键主链路测试
- 错误码一致性检查

### 5.2 高风险改动加跑测试
以下改动必须额外跑完整兼容回归：
- BookSource 模型变更
- 规则执行器变更
- 网络层变更
- Cookie / 登录能力变更
- JS 相关能力变更
- 失败分类与错误码变更

## 6. 回归规则
### 6.1 样本绑定要求
每个兼容性改动必须：
- 绑定至少 1 个样本
- 说明该样本之前失败原因
- 说明此次改动后的预期结果

### 6.2 回归结果输出要求
回归报告必须至少包含：
- 总样本数
- 通过数
- 失败数
- 新增通过数
- 新增失败数
- 失败原因分布
- 是否影响兼容率

### 6.3 禁止事项
- 不允许“修一个样本，打坏一批样本”但不说明
- 不允许只人工点测，不跑样本回归
- 不允许修改兼容口径但不更新文档

## 7. 错误码与日志规则
- 错误码应稳定、可枚举、可搜索
- 日志必须标记失败阶段
- 日志必须能关联样本或书源
- 错误码变化必须在 PR 中说明

## 8. 回退机制
以下情况必须准备明确回退方案：
- Core 主流程改动
- 数据结构改动
- 网络层改动
- Cookie / 登录能力改动
- JS 支持策略改动

回退方案至少包括：
- 回退条件
- 回退方式
- 回退后影响面
- 是否需要关闭功能开关

## 9. 文档同步规则
以下变更必须同步更新文档：
- 兼容规范变更
- 能力支持矩阵变更
- 错误码新增或变更
- 样本目录与分类规则变更
- 门禁规则变更

## 10. 发布前门禁
进入发布候选版本前，必须满足：

- 主链路样本回归通过
- 无阻断级错误
- 高优先级失败原因已评估
- 兼容率达到阶段目标
- 文档已同步
- 回退预案可执行

## 11. 阶段性验收口径
### PoC 阶段
- 非 JS 主链路闭环可跑通
- 样本回归机制已建立
- 错误定位可用

### MVP 阶段
- 首版能力矩阵实现到位
- 兼容率达成目标
- 调试页可支撑问题定位
- 门禁流程可执行
